# Отчет об оптимизации логики спавна и задача по синхронизации голоса

## Внесенные изменения: Отключение Smart Spawn для Tutorial
### Проблематика
В сцене обучения `Tutorial.unity` отсутствуют компоненты генерации мира (например, `NetworkWorldGenerator`), которые необходимы для работы алгоритма "умного" поиска безопасной точки спавна. Попытка вызвать серверный RPC для поиска безопасного места в этой сцене приводила к логическим ошибкам или потенциальным `NullReferenceException`.

### Реализованное решение
Логика отключена на стороне клиента, чтобы не нагружать сервер лишними проверками сцен:
1.  **`PlayerBehaviour.SetLoadedPosition()`**: Добавлена проверка `GameManager.IsTutorialScene()`. В случае обучения игрок устанавливается в начальную позицию локально.
2.  **`NetworkPlayer.Client_Started()`**: Проверка `IsTutorialScene()` предотвращает отправку `RequestSafeSpawnServerRpc` при старте.
Это гарантирует, что в туториале игрок всегда появляется в строго заданной точке, а в основном мире (`World.unity`) продолжает работать система безопасного спавна.

## План на будущее: Оптимизация отправки голосовых данных (Voice Sync)

### Обнаруженная проблема
В логах сервера наблюдается предупреждение:
`[Netcode] Deferred messages were received for a trigger of type OnSpawn with key 0, but that trigger was not received within 10 second(s).`
Это происходит, когда сервер начинает отправлять голосовые клипы (`SendVoicesServerRpc`) клиенту, который еще не успел полностью инициализировать (проспавнить) свои сетевые объекты (в частности, глобальный объект `[# GAME #]` с ID 0).

### Технические детали
*   **Причина:** Преждевременная инициализация передачи тяжелых данных (аудио) до завершения фазы Network Spawn у клиента.
*   **Риски:** Потеря пакетов данных о голосе в первые секунды подключения, избыточное потребление памяти Netcode на хранение "отложенных" сообщений.

### Предлагаемое решение
Необходимо перенести запуск отправки голоса (`StartSendNewsVoice`) с события подключения (`OnClientConnected`) на событие полной регистрации пользователя.
*   **Файл:** `Assets/Scripts/Multiplayer/Release Notes/AudioClipSender.cs`
*   **Метод решения:** Подписаться на `NetworkUserManager.onUserRegistred`. Начинать передачу только после того, как сервер подтвердит успешную регистрацию имени игрока. Это гарантирует, что все базовые `NetworkObject` уже существуют на обоих концах.
